# 样本流转链路登记工具

实验样本全生命周期追踪系统，支持从采集、入库、出库、检测到归档的完整交接过程记录。

## 技术栈

- **前端**: React 18 + TypeScript + Vite
- **状态管理**: Zustand
- **样式**: TailwindCSS 3
- **图标**: Lucide React
- **本地存储**: IndexedDB (idb 库)
- **CSV 解析**: Papa Parse
- **后端**: Express 4 (可选，静态服务)

## 启动方式

### 前置要求
- Node.js >= 18
- npm 或 pnpm

### 安装依赖

```bash
npm install
```

### 启动开发服务器

```bash
# 同时启动前端和后端（推荐）
npm run dev

# 仅启动前端
npm run client:dev

# 仅启动后端
npm run server:dev
```

启动后访问: http://localhost:5173

### 构建生产版本

```bash
npm run build
```

## 快速验证指南（按顺序执行）

### 🎯 第1步：登录系统

使用以下任一预设账户登录，默认密码均为 **123456**：

| 用户名 | 角色 | 用途 |
|--------|------|------|
| `admin` | 管理员 | 全功能验证 |
| `collector01` | 采集员 | 批次导入验证 |
| `warehouse01` | 库管员 | 入库/出库验证 |
| `tester01` | 检测员 | 检测流程验证 |
| `auditor01` | 审核员 | 归档/回退/导出验证 |

> 💡 登录页有快速切换角色的快捷按钮

---

### 🎯 第2步：导入样例批次（采集员或管理员）

1. 登录为 `collector01` 或 `admin`
2. 进入「批次导入」页面
3. 使用项目根目录 `samples/` 下的样例文件：
   - `samples/batch-demo-001.csv`（CSV 格式，8条数据）
   - `samples/batch-demo-002.json`（JSON 格式，5条数据）
4. 填写批次号（如 `BATCH-TEST-001`）和备注
5. 点击「确认导入批次」
6. ✅ 验证点：成功导入后显示导入数量和批次号

---

### 🎯 第3步：入库登记（库管员）

1. 登录为 `warehouse01`
2. 进入「入库登记」页面
3. 选择一个待入库样本
4. 选择目标存储库位（如 `WH-A-01`）
5. 点击「确认入库」
6. ✅ 验证点：
   - 样本状态从「待入库」变为「在库」
   - 当前库位正确显示
   - 当前持有人为库管员

---

### 🎯 第4步：出库交接（库管员 → 检测员）

1. 保持登录 `warehouse01`
2. 进入「出库交接」页面
3. 选择一个「在库」样本
4. **重要**：转出确认库位必须与样本当前实际库位一致
5. 选择检测接收人（如 `tester01 / 王检测`）
6. 点击「确认出库交接」
7. ✅ 验证点：
   - 样本状态变为「送检中」
   - 持有人变更为接收的检测员

---

### 🎯 第5步：检测接收（检测员）

1. 切换登录为 `tester01`
2. 进入「检测接收」页面
3. 选择送检中的样本
4. 选择检测区域库位（如 `LAB-01`）
5. 点击「确认接收」
6. ✅ 验证点：
   - 样本状态变为「检测中」
   - 库位变为检测区域

---

### 🎯 第6步：检测完成（检测员）

1. 保持登录 `tester01`
2. 进入「检测完成」页面
3. 选择检测中的样本
4. 填写检测结果（如「阴性/正常」）
5. 点击「提交检测完成」
6. ✅ 验证点：
   - 样本状态变为「检测完成」

---

### 🎯 第7步：复核归档（审核员）

1. 切换登录为 `auditor01`
2. 进入「归档复核」页面
3. **先在「样本复核」标签页**：选择检测完成的样本，点击「确认复核」
4. ✅ 验证点：样本被标记为「已复核」
5. **再切换到「最终归档」标签页**：选择已复核样本，选择归档库位（`ARC-01`），点击「确认归档」
6. ✅ 验证点：
   - 样本状态变为「已归档」
   - 样本被锁定，无法再编辑

---

### 🎯 第8步：导出审计时间线（审核员）

1. 保持登录 `auditor01`
2. 进入「审计导出」页面
3. 选择导出格式（CSV 或 JSON）
4. 可选：设置日期、操作者、操作类型筛选
5. 点击「导出审计数据」
6. ✅ 验证点：
   - 文件成功下载
   - CSV/JSON 包含完整流转记录、操作者、时间戳、回退记录等

## ⚠️ 异常校验测试场景

以下操作**应该被拒绝**，请逐一验证：

### ❌ 1. 重复样本号导入
- 尝试导入已存在的样本号，系统应提示重复并拒绝

### ❌ 2. 从错误库位转出
- 出库时，故意选择与样本实际库位不一致的库位
- 系统应拒绝出库并提示「样本当前所在库位与选择的转出库位不一致」

### ❌ 3. 未复核就归档
- 跳过复核步骤，直接尝试归档一个未复核的「检测完成」样本
- 系统应拒绝并提示「样本未经过审核员复核，无法归档」

### ❌ 4. 已归档样本普通编辑
- 对已归档样本尝试执行入库/出库等操作
- 系统应拒绝并提示「样本已归档，禁止执行任何编辑或流转操作」

### ❌ 5. 回退不存在的交接
- 在异常回退页面，系统只允许选择存在且未被回退过的交接记录
- 尝试回退不存在的 ID 会被拒绝

---

### 🎯 第9步：异常回退与数据完整性验证

1. 登录为 `auditor01`
2. 进入「异常回退」页面
3. 选择一个样本，回退最近一次交接操作（如归档）
4. ✅ 验证点：
   - 样本状态恢复到上一状态
   - 当前位置、持有人正确恢复
5. **刷新浏览器页面**（模拟重启）
6. 重新登录，进入「审计时间线」或样本详情页
7. ✅ 验证点（数据完整性）：
   - 样本当前位置正确保留
   - 失败交接记录完整保留
   - 回退记录（操作者、原因、时间）完整保留
   - 导出的审计链路包含所有历史记录

## 数据存储说明

所有数据存储在浏览器本地 IndexedDB（数据库名：`SampleTrackingDB`），包含：

- **users**: 用户账户（含密码哈希）
- **batches**: 批次信息
- **samples**: 样本主数据
- **locations**: 库位配置
- **transferRecords**: 完整流转记录（含回退标记）
- **failedTransfers**: 失败交接记录
- **auditLogs**: 系统操作审计日志

> 数据完全本地化，不会上传到任何服务器。清除浏览器数据会丢失所有记录。
